# Fortune Planning V3 - 宜忌模块后端重构开发指南

## 1. 背景与目标
当前“宜忌”模块的 AI 吐出内容过于冗长且缺乏强行动指令。
本次 V3 重构的核心目标是：**将宜忌模块从“解释型阅读”升级为“指令型行动（Action-oriented）”**，并在前后端分离的架构下引入**多级漏斗路由**和**强结构化 JSON 输出**。

前端现已完成降级与极客模式（术语截断）的 UI 适配（纯展示层）。现需后端对 API 的 `do` 和 `dont` 数组输出结构进行对齐与数据清洗。

## 2. 核心数据结构约束 (数据模型约定)

后续后端返回的 AI 内容中，`do` 与 `dont` 数组必须严格符合以下 `PaidYiJiItem` 结构，并且流日（Daily）级别支持顶层的 `note` 字段。

### 2.1 JSON Schema 规范

```typescript
export type PaidYiJiItem = {
  action: string;           // 核心指令：必须动词开头，极简（例如：“果断推进签约”）
  reason?: string;          // 支撑逻辑：白话文解释 + 【玄学依据：专业术语】
  tags?: string[];          // 场景标签：用于前端分类（如：职业、财务、人际、健康）
  priority?: 1 | 2 | 3;     // 优先级：1为今日关键行动，2为次要，3为补充
  source?: 'template' | 'ai-extract' | 'legacy'; // 溯源字段：用于埋点追踪和A/B测试
};

export type PaidYiJiResult = {
  dimension: 'year' | 'month' | 'day'; 
  do: PaidYiJiItem[];
  dont: PaidYiJiItem[];
  note?: {                  // 特别提醒（如天刑/白虎，仅限流日或特殊高敏流月触发）
    level: 'warning' | 'danger'; 
    content: string;
  };
};
```

### 2.2 字段深度解析与规则要求

1. **`action` (强指令，极度精简)**
   - **规则**：**必须以动词开头**。绝不能包含“建议”、“适合”、“应当避免”、“忌”等前缀废话。
   - **限制**：字数严格控制在 **20字以内**。
   - **错误示例**：“建议你今天可以去尝试拓展新客户” (❌)
   - **正确示例**：“主动拜访并拓展新客户” (✅)

2. **`reason` (结构化支撑逻辑，支持前端极客模式)**
   - **规则**：包含白话文解释 + `【玄学依据：xxx】` 的组合格式。
   - **原因**：前端已实现无缝正则截断。如果用户未开启“极客模式”，前端会自动切掉 `【` 之后的内容。大模型/模板吐词时，**必须把术语/星曜解释关进 `【玄学依据：...】` 括号里**。
   - **正确示例**：“简单的知识分享能提升信任权重。【玄学依据：太阳具备施予特性】”

3. **`priority` (前端视觉分层)**
   - **1**：高优/核心指令（前端大字号平铺展示）。
   - **2 / 3**：补充建议（前端将被放入折叠面板收纳）。

## 3. 后端生成策略与护栏 (Guardrails)

为了防范大模型（LLM）的“幻觉”和“过度啰嗦”，后端在接收到 LLM 吐出 JSON、落库并返回给前端之前，**必须在 Service 层加上一道清洗脚本 (Middleware)**：

### 护栏 1：动词强制裁剪与超长剥离
- 若 `action` 超过 20 个字，后端应脚本介入：截断前 20 字作为 `action`，并自动把超出的后半段拼接到 `reason` 的开头。
- 用正则清洗掉 `action` 里的前缀（如去除“建议/适合/可以/应当避免/避免/不要”等）。

### 护栏 2：忌中带宜 (防串岗)
- 遍历 `dont` 数组，如果 `action` 包含了“宜”、“适合”、“利于”等正向词汇，直接记录 Log 并在服务端进行替换或丢弃（前端红色卡片里绝对不能出现绿色语义）。

### 护栏 3：同主题合并 (防止刷列表感)
- 如果 `do` 或 `dont` 数组中出现多条 `tags` 相同（例如都属于“财务”）的建议，后端脚本应按 `priority` 排序，只保留最高优先级的作为独立 Item，其余的内容提取并拼接到该条的 `reason` 中（例：“此外，也可考虑…”），以降低前端的信息密度。

## 4. 多级漏斗路由 (Multi-level Template Routing)
除了兜底的大模型提取逻辑，建议后端引入漏斗式匹配机制（基于盘局特征直接下发静态 JSON 模板，降低大模型并发与延迟）：

1. **Level 1（星曜画像主模板）**：例如命中“命宫太阳星”，直接吐出太阳星高优模板（包含 do/dont）。
2. **Level 2（阶段四化触发模板）**：若无 L1，看本年/月是否有核心星化禄/化忌触发。
3. **Level 3（通用宫位+主题模板）**：例如“流日在迁移宫 + 职业主题”。
4. **Fallback（大模型兜底）**：兜底当前 AI 生成与清洗逻辑（过上述护栏）。

*注：流日的特殊星曜（如天刑/白虎）命中时，触发顶层 `note` 字段预警。*
